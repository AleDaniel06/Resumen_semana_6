¿QUÉ ES UNA
BASE DE DATOS?

📖 DEFINICIÓN

Conjunto organizado de datos almacenados electrónicamente. Permite la recuperación, modificación y administración eficiente y segura de la información.

SQL Server organiza todo en tablas (filas × columnas) — igual que Excel pero con superpoderes.

🗂️ TIPOS DE BD

📊
Relacionales
SQL · tablas
📄
NoSQL
sin esquema
📁
Documentos
flexibles
🔗
Grafos
nodos y aristas

🔑 CONCEPTOS CLAVE

📋 COLUMNAS (columns)

Almacenan datos. Cada columna tiene tipo de dato y nombre único. Ej: nombre (cadena), fecha_nac (fecha), hijos (int).

🔑 CLAVE PRIMARIA (PK)

Valor único e irrepetible que identifica cada fila. No puede ser nulo. Generalmente la primera columna.

🔗 CLAVE FORÁNEA (FK)

Referencia la PK de otra tabla. Crea relaciones entre tablas y garantiza integridad referencial.

📊 TIPOS DE DATOS

Cada columna solo acepta un tipo: varchar, int, date, bit, varbinary, etc. Define el almacenamiento.

// NIVEL 02 · SISTEMA

BASES DE DATOS
DEL SISTEMA

Al instalar SQL Server se crean automáticamente 4 bases de datos del sistema. Contienen la metadata necesaria para operar y administrar el servidor.

🏰
master

La BD principal del sistema. Controla usuarios y operaciones del servidor: cuentas, variables de entorno y mensajes de error. Sin ella, SQL Server no puede arrancar.

📐
model

Plantilla o molde para nuevas BD. Cuando se crea una BD, todo el contenido de model se copia en la nueva. Permite personalizar el proceso de creación.

📅
msdb

Utilizada por el SQL Server Agent para programar alertas y trabajos, y para registrar operadores. Gestiona tareas automatizadas del sistema.

⚡
tempdb

Para almacenamiento temporal. Se recrea cada vez que se inicia SQL Server. Guarda tablas temporales, variables de tabla y resultados intermedios.

📊 BD DE USUARIO VS BD DE SISTEMA

⚙️ BD SISTEMA

Creadas automáticamente. Contienen tablas de sistema con metadata para operar y administrar el servidor. No se deben modificar manualmente.

👤 BD USUARIO

Las que nosotros creamos. Almacenan nuestros datos. También contienen tablas de sistema propias con metadata de nuestra BD.

// NIVEL 03 · CREACIÓN

CREAR UNA
BASE DE DATOS

Una BD en SQL Server está formada por mínimo 2 archivos físicos. Pueden existir archivos adicionales secundarios y de log.

💾
PRIMARIO
.mdf
Archivo principal. Almacena tablas del sistema y puede almacenar tablas de usuario. Obligatorio, solo uno por BD.
📂
SECUNDARIO
.ndf
Archivos adicionales para tablas de usuario. Útil cuando la BD es muy grande. Pueden estar en discos diferentes para mejor rendimiento.
📜
TRANSACTION LOG
.ldf
Registra todas las transacciones. Mecanismo de seguridad para recuperar la BD ante fallos. Al menos uno por BD.

⌨️ SINTAXIS CREATE DATABASE

CREATE DATABASE nombre_basedatos
ON [PRIMARY] (
  NAME = nombre_lógico_data ,
  FILENAME = 'ruta\archivo.mdf' ,
  SIZE = 8MB ,
  MAXSIZE =
    15MB ,
  FILEGROWTH = 1MB )
LOG ON (
  NAME = nombre_lógico_log ,
  FILENAME = 'ruta\archivo.ldf' ,
  SIZE = 3MB ,
  MAXSIZE =
    8MB ,
  FILEGROWTH = 10% )
GO
Ejercicio 2.1: Crear con propiedades predeterminadas:
CREATE DATABASE DBPrueba1 · GO
Luego verificar en el Explorador de objetos → Bases de datos → DBPrueba1

📋 PARÁMETROS CLAVE

NAME
Nombre lógico usado en sentencias SQL
FILENAME
Ruta completa y nombre del archivo físico
SIZE
Tamaño inicial del archivo
MAXSIZE
Tamaño máximo (o UNLIMITED)
FILEGROWTH
Incremento cuando necesita más espacio (KB/MB/%)
ON PRIMARY
Propiedades del archivo primario (.mdf)
LOG ON
Propiedades del archivo de log (.ldf)

// NIVEL 03B · TRANSACCIONES

TRANSACCIONES
Y LOG

🔄 ¿QUÉ ES UNA TRANSACCIÓN?

Un conjunto de operaciones de modificación de datos que se ejecutan como una sola unidad. Si una operación falla, toda la transacción se anula (rollback).

Garantiza que las operaciones se llevarán a cabo completas o no se realiza ninguna, protegiendo la consistencia de los datos.

📜 REGISTRO DE TRANSACCIONES

El archivo .ldf registra todas las operaciones INSERT, UPDATE y DELETE. El proceso:

① Aplicación envía modificación al servidor SQL
② Páginas afectadas localizadas en el caché
③ Modificación registrada en el transaction log
④ Checkpoint guarda transacciones confirmadas en BD
Si el sistema falla, la recuperación automática usa el log para restaurar todas las transacciones confirmadas y anular las incompletas.

🎮 EJEMPLO DE TRANSACCIÓN

Registrar la venta de 10 unidades del producto ABC (tablas: Factura, Detalle_factura, Producto):

1
Registra en Factura la cabecera de la venta
✓ OK
2
Registra en Detalle_factura los detalles
✓ OK
3
Actualiza Producto reduciendo el stock → FALLA
✗ FAIL
↩
ROLLBACK: se anulan las operaciones 1 y 2
⚠ ROLLBACK
Sin transacciones: Factura y Detalle_factura quedarían registrados pero el stock no se actualizaría → inconsistencia de datos.

20 preguntas teóricas y prácticas del Capítulo II resueltas con detalle técnico.

◈ PREGUNTAS TEÓRICAS
10 MISIONES
1En control académico, ¿por qué es crítico definir correctamente claves primarias y foráneas en estudiantes, cursos y matrículas?
Las claves primarias (PK) garantizan que no existan registros duplicados de un estudiante o curso. Las claves foráneas (FK) aseguran integridad referencial: una matrícula no puede existir sin un estudiante válido ni un curso válido. Sin esta correcta definición, se producirían inconsistencias como matrículas huérfanas (sin estudiante o sin curso), duplicaciones de datos y reportes de calificaciones incorrectos, comprometiendo la confiabilidad del sistema académico completo.
2¿Stored procedures o funciones definidas por el usuario para centralizar reglas de negocio? ¿Qué criterios considerar?
Los stored procedures son más adecuados para operaciones con múltiples pasos (INSERT, UPDATE, DELETE, control de transacciones) y lógica de negocio compleja. Las funciones son ideales para cálculos reutilizables dentro de consultas SELECT. Criterios: (1) Si se necesita retornar un valor único y usarlo en SELECT → función. (2) Si implica múltiples operaciones DML o manejo de errores → stored procedure. (3) Para auditorías y control transaccional → stored procedure.
3En e-commerce, ¿qué rol cumplen los índices y estadísticas al manejar búsquedas de productos?
Los índices aceleran las búsquedas de productos evitando escaneos completos de tabla (como buscar en el índice de un libro en vez de leer página por página). Para un e-commerce con millones de productos, un índice en columnas como nombre_producto, categoria o precio reduce el tiempo de consulta de segundos a milisegundos. Las estadísticas informan al optimizador de SQL Server sobre la distribución de valores, permitiéndole elegir el plan de ejecución más eficiente para cada búsqueda.
4¿Qué ventajas tendría usar vistas en un sistema financiero con información sensible?
Las vistas permiten: (1) Seguridad: ocultar columnas sensibles (salarios, cuentas bancarias) mostrando solo las permitidas a cada usuario. (2) Simplificación: encapsular joins complejos entre tablas financieras en una consulta simple. (3) Abstracción: los usuarios consultan la vista sin conocer la estructura física de las tablas subyacentes. (4) Control de acceso: se otorgan permisos de SELECT sobre la vista, no sobre las tablas directamente.
5Si el archivo de transaction log se llena constantemente en una BD de ventas, ¿qué consecuencias tendría para la consistencia?
Si el transaction log (.ldf) se llena: (1) SQL Server detiene todas las operaciones de escritura (INSERT, UPDATE, DELETE) hasta que haya espacio. (2) Los registros de ventas pendientes no se pueden completar, generando errores en la aplicación. (3) En caso de fallo durante este estado, las transacciones incompletas no podrían registrarse correctamente. Solución: configurar FILEGROWTH adecuado, realizar backups de log regularmente con BACKUP LOG, y monitorear el crecimiento del archivo.
6¿Qué impacto tendría habilitar autoshrink en una BD de producción?
Habilitar autoshrink en producción es generalmente perjudicial: (1) SQL Server comprime automáticamente los archivos de BD cuando no se usan al 100%, liberando espacio. (2) Sin embargo, esto fragmenta los índices, degradando el rendimiento de las consultas. (3) Si la BD crece de nuevo, SQL Server debe reasignar espacio, generando operaciones costosas. (4) Produce ciclos de shrink-grow que consumen CPU y E/S. Recomendación: mantener autoshrink en false y gestionar el espacio manualmente con DBCC SHRINKFILE.
7En un sistema hospitalario, ¿qué pasaría si no se aplican restricciones CHECK en la tabla de pacientes para validar que la edad sea positiva?
Sin restricción CHECK (edad > 0): (1) Se podrían registrar pacientes con edades negativas o cero, generando datos inválidos. (2) Los reportes estadísticos (promedio de edad, distribución etaria) serían incorrectos. (3) Los sistemas de alerta por rango de edad (ej: vacunas pediátricas) fallarían. (4) Auditorías de calidad de datos detectarían inconsistencias. La restricción CHECK actúa como validación a nivel de BD, independientemente de la aplicación que inserte los datos.
8¿Por qué la BD model es fundamental al crear nuevas bases de datos en un proyecto empresarial?
La BD model actúa como plantilla: toda nueva BD hereda automáticamente su contenido y configuración. En un proyecto empresarial, esto permite: (1) Estandarización: incluir en model tablas de auditoría, usuarios estándar o configuraciones comunes que toda BD empresarial debe tener. (2) Consistencia: todas las BD del proyecto tendrán las mismas opciones de configuración (collation, recovery model, etc.). (3) Eficiencia: no hay que configurar manualmente cada nueva BD desde cero.
9¿Cómo afectaría un mal diseño de transacciones en una aplicación bancaria al ejecutar transferencias entre cuentas?
Una transferencia involucra: débito en cuenta origen + crédito en cuenta destino. Sin correctas transacciones: (1) Si el débito se ejecuta pero el crédito falla, el dinero desaparece del sistema. (2) Múltiples usuarios ejecutando transferencias simultáneas sin control transaccional producirían condiciones de carrera (race conditions) con saldos incorrectos. (3) Un rollback parcial dejaría la BD en estado inconsistente. Con transacciones ACID correctas, ambas operaciones se ejecutan juntas o ninguna, garantizando que el dinero total en el sistema permanezca constante.
10¿Qué riesgos implica configurar una BD crítica en single_user durante la implementación de un ERP?
Configurar en single_user durante la implementación de un ERP implica: (1) Solo un usuario puede conectarse a la vez — si el DBA ocupa la conexión, el equipo de implementación no puede acceder. (2) Si la implementación del ERP requiere múltiples conexiones simultáneas (procesos de migración, validación, pruebas paralelas), se bloqueará. (3) En producción, los usuarios finales no podrán acceder durante el tiempo que dure la operación de mantenimiento. Alternativa recomendada: usar RESTRICTED_USER que permite solo administradores y propietarios, o planificar ventanas de mantenimiento.

5Diseña una transacción que registre una venta en Factura, DetalleFactura y actualice el Stock de productos.
BEGIN TRANSACTION
  BEGIN TRY
    -- Operación 1: Cabecera de la factura
    INSERT INTO Factura (idFactura, fecha, idCliente)
    VALUES (1001, GETDATE(), 5)

    -- Operación 2: Detalle de la venta
    INSERT INTO DetalleFactura (idFactura, idProducto, cantidad)
    VALUES (1001, 42, 10)

    -- Operación 3: Actualizar stock
    UPDATE Producto SET stock = stock - 10
    WHERE idProducto = 42

    COMMIT TRANSACTION
  END TRY
  BEGIN CATCH
    ROLLBACK TRANSACTION
    PRINT 'Error: transacción revertida'
  END CATCH
GO
